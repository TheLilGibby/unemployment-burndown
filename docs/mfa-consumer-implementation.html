Consumer Multi-Factor Authentication (MFA)

Plaid Security Questionnaire — Question 4 Supporting Documentation
Prepared: March 3, 2026 · Application: Financial Burndown Tracker

Summary
RAG Consulting LLC provides TOTP-based multi-factor authentication (MFA) for all consumers on the Financial Burndown Tracker web application. MFA is available to all users and is verified before Plaid Link is surfaced. Users must authenticate (including MFA verification when enabled) before accessing any financial features or connecting bank accounts.

2. MFA Setup Flow

Users enable MFA from the Settings page within the application. The setup process is as follows:

  1. Initiate Setup: Authenticated user clicks "Set Up 2FA" in the Settings panel.
  2. Secret Generation: The backend generates a cryptographic TOTP secret using otplib authenticator.generateSecret().
  3. QR Code Display: A QR code is generated and displayed to the user, encoding the TOTP URI (otpauth://totp/BurndownTracker:{email}?secret={secret}).
  4. Manual Entry Option: The raw secret key is also displayed for users who prefer manual entry into their authenticator app.
  5. Verification: The user enters the 6-digit code from their authenticator app.
  6. Activation: The backend verifies the code against the secret using authenticator.verify(). On success, MFA is permanently enabled for the account and the secret is stored securely.
  7. Audit Logging: MFA enablement is recorded in the structured audit log.

3. MFA Login Flow

When a user with MFA enabled attempts to log in:

  1. Credential Verification: The user submits their email and password. Passwords are verified against bcrypt hashes (12 rounds).
  2. MFA Challenge: If credentials are valid and MFA is enabled, the server returns mfaRequired: true along with a temporary JWT token. This temporary token has mfaVerified: false and cannot be used to access any protected resources or financial data.
  3. Code Entry: The user is presented with a dedicated MFA verification screen requesting their 6-digit authenticator code.
  4. Code Verification: The backend verifies the submitted TOTP code against the user's stored secret using authenticator.verify().
  5. Full Access Token: On successful verification, a new JWT is issued with mfaVerified: true. Only this token grants access to the application dashboard and financial features.
  6. Audit Logging: Both successful and failed MFA verification attempts are recorded in the audit log.

4. MFA Enforcement Before Plaid Link

The application enforces a strict authentication gate before any financial features are accessible:

  • Unauthenticated users see only the Login/Register screen. No financial data or Plaid Link is rendered.
  • Users with MFA enabled must complete the full MFA verification step during login before reaching the application dashboard.
  • The "Connect Bank" button (which launches Plaid Link) is only rendered within the authenticated dashboard, behind the login + MFA gate.
  • Before Plaid Link opens, users are additionally presented with a consent disclosure modal that describes the data being accessed and requires explicit agreement.
  • The Plaid link token endpoint (/plaid/link-token) requires a valid JWT with organization membership, which itself requires successful authentication (including MFA if enabled).
Authentication Flow to Plaid Link
Login → MFA Verification (if enabled) → Dashboard → "Connect Bank" → Consent Modal → Plaid Link

5. Backend Authorization Middleware

All API endpoints that handle financial data use the requireAuth() middleware, which:

  • Extracts and validates the JWT Bearer token from the Authorization header.
  • Rejects requests with missing, invalid, or expired tokens (HTTP 401).
  • Supports an optional requireMfa: true flag that rejects tokens where mfaVerified is false (HTTP 403).

Additionally, the requireOrg() middleware ensures users belong to an organization before accessing Plaid-related endpoints, providing data isolation between users.

8. Security Controls Summary

  • Anti-Enumeration: Login and password reset endpoints return generic error messages that do not reveal whether an email exists in the system.
  • Session Isolation: Tokens stored in sessionStorage are not accessible across tabs or after browser close.
  • Audit Trail: All authentication events (login, MFA setup, MFA verification, failed attempts) are logged in structured JSON format with timestamps and user identifiers.
  • Sensitive Data Redaction: Passwords, tokens, and secrets are automatically redacted from all log output.
