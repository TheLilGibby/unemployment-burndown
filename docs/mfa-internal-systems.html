Multi-Factor Authentication for Critical Systems

Plaid Security Questionnaire — Question 5 Supporting Documentation
Prepared: March 3, 2026 · Application: Financial Burndown Tracker

Summary
Multi-factor authentication (MFA) is required for all personnel accessing critical systems that store or process consumer financial data. This includes the AWS Console, GitHub source code repositories, and production infrastructure. This requirement is mandated by Section 6.2 of the RAG Consulting LLC Information Security Policy (v1.0, effective February 27, 2026).

2. Policy Basis

The MFA requirement is defined in the RAG Consulting LLC Information Security Policy, Section 6.2 — Authentication:

"Multi-factor authentication (MFA) is required for:
  • AWS Console access
  • GitHub repository access
  • Production infrastructure access
Service accounts use scoped IAM roles with temporary credentials where possible."

3. AWS Infrastructure Security

3.1 IAM and Access Control

  • Least Privilege: All IAM users and roles are granted only the minimum permissions required for their function.
  • Role-Based Access Control (RBAC): Access is assigned based on job function through AWS IAM roles and policies.
  • Scoped Lambda Execution Roles: Each Lambda function has a dedicated IAM role with policies limited to the specific DynamoDB tables, S3 buckets, and AWS services it requires (e.g., DynamoDBCrudPolicy, S3CrudPolicy).
  • No Wildcard Policies: IAM policies specify explicit resource ARNs rather than wildcard (*) permissions.

3.3 Secrets Management

  • Application secrets (Plaid API keys, JWT secret, encryption keys, SendGrid API key) are stored as encrypted environment variables in AWS Lambda using NoEcho CloudFormation parameters.
  • Secrets are never committed to source code repositories.
  • API keys and tokens are rotated on a defined schedule or immediately upon suspected compromise.

4. Access Review Process

Per Section 6.3 of the Information Security Policy:

  • Quarterly Reviews: Access to production systems and sensitive data stores is reviewed quarterly.
  • Prompt Revocation: Unused or excessive permissions are revoked promptly upon identification.
  • Separation of Duties: Critical operations require multiple approvals where feasible.

5. CI/CD Pipeline Security

  • All code changes require pull request review before merging to the main branch.
  • CI pipeline (GitHub Actions) runs automated checks: linting, tests, and production build verification on every push and pull request.
  • GitHub Dependabot performs weekly automated dependency scanning across all package ecosystems.
  • GitHub access requires MFA, ensuring that only authenticated and MFA-verified personnel can approve or merge code changes.
